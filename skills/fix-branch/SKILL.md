# Execute Fix Plan

Apply fixes from a previously generated fix plan for branch `$ARGUMENTS`.

## Step 0: Validate

If no branch name was provided, ask the user.

Ensure the directory exists:
```bash
mkdir -p ./claude-context/$ARGUMENTS
```

Check that the fix plan exists:
```bash
cat ./claude-context/$ARGUMENTS/fix-plan.md
```

If it doesn't exist, inform the user to run `/review-branch $ARGUMENTS` first.

Also read the review for full context:
```bash
cat ./claude-context/$ARGUMENTS/review.md
```

## Step 1: Parse and present the fix plan

Read `./claude-context/$ARGUMENTS/fix-plan.md` and present all fixes as a numbered list with severity:
- Show: Fix #, Title, Severity, Repo, Files affected
- Ask the user which fixes to apply (all, specific numbers, or by severity level)

## Step 2: Load repo config

Read `./claude-context/review-config.json` to get repo paths.

## Step 3: Apply fixes one-by-one

For each selected fix, in priority order:

### 3a. Checkout the branch
```bash
cd <repo-path> && git checkout $ARGUMENTS && git pull origin $ARGUMENTS
```

### 3b. Read the current state of affected files
Before making any changes, read ALL files listed in the fix to understand current code.

### 3c. Apply the fix — Senior Engineer Standards

Follow the concrete steps from the fix plan. For each change:
- Read the file first AND its surrounding context (callers, related classes, specs)
- Understand the WHY before changing anything
- Apply the minimal change needed — no gold-plating

**SOLID — enforce on every fix:**
- **S** Single Responsibility: One fix = one concern. If a method/class does too much, extract. Never stuff logic into an existing method just because it's convenient.
- **O** Open/Closed: Extend behavior via new methods/classes, don't mutate existing contracts that other code depends on. Prefer composition over modification.
- **L** Liskov Substitution: Subclasses/implementations must honor the contract. If you change a return type or add a required param, check ALL callers.
- **I** Interface Segregation: Don't force a class to depend on methods it doesn't use. Keep interfaces focused.
- **D** Dependency Inversion: Depend on abstractions. Inject dependencies, don't hardcode. Use config/env for external services.

**Clean Code — enforce on every fix:**
- Methods should do ONE thing and be < 20 lines. Extract if longer.
- Naming must be intention-revealing. No abbreviations, no generic names (data, info, result, tmp).
- No magic numbers/strings — use constants or config.
- No deep nesting (max 2 levels). Use early returns / guard clauses.
- No dead code, no commented-out code. Delete it.
- DRY — but don't over-abstract. 3+ duplications = extract, 2 = leave it.

**Maintainability — enforce on every fix:**
- Every public method should have clear input/output contracts.
- Error handling must be explicit — no silent rescues, no bare `rescue => e` without logging.
- Log with context (who, what, why) at appropriate levels.
- If the fix touches a hot path, consider performance implications.
- If the fix involves state changes, ensure idempotency.

**ACID — enforce on DB-related fixes:**
- **A** Atomicity: Wrap related changes in a transaction. If one part fails, ALL roll back.
- **C** Consistency: Validate data invariants. Add DB constraints where possible, not just app-level checks.
- **I** Isolation: Use proper locking (`FOR UPDATE`, advisory locks) for concurrent access. Check for race conditions.
- **D** Durability: Ensure changes persist. Don't rely on in-memory state for critical data.

**Security — check on every fix:**
- No SQL injection (use parameterized queries)
- No mass assignment (whitelist params)
- Validate/sanitize all external input
- Don't expose internal errors to users

**Optimization — check on every fix:**
- N+1 queries: Use eager loading (`eager`, `eager_graph`, `includes`). If a loop calls DB, refactor to batch.
- Unnecessary queries: Don't load full records when you only need a count or specific columns. Use `select_map`, `pluck`, `count`.
- Index awareness: If adding a `WHERE` clause or `ORDER BY`, verify an index exists. Suggest adding one if missing.
- Memory: Don't load entire tables into memory. Use `paged_each` / `find_each` for large datasets. Stream instead of buffer.
- Caching: For repeated expensive lookups, consider memoization (`||=`) or cache (Redis). But only where invalidation is simple.
- Background jobs: Move slow operations (emails, external API calls, heavy computation) to Sidekiq. Don't block the request cycle.
- Serialization: Don't serialize more data than the client needs. Keep API responses lean.
- Connection pooling: Don't open new connections in loops. Reuse DB/HTTP connections.
- Hot paths: If the fix touches code called on every request (middleware, auth, logging), benchmark before and after.

### 3d. Self-review before committing

Before considering a fix done, review your own changes as a critical senior engineer:

1. **Read the full diff** of what you changed:
```bash
cd <repo-path> && git diff
```
2. Ask yourself:
   - Would I approve this in a PR review?
   - Is there a simpler way to achieve the same result?
   - Did I introduce any new code smells?
   - Are there edge cases I missed?
   - Is this change backward-compatible?
3. If the answer to any is "no" — fix it before proceeding.

### 3e. Verify the fix

After each fix:
1. Run relevant tests if they exist:
```bash
cd <repo-path> && <test-command>
```
2. Check for syntax errors / linting
3. Confirm the fix addresses the original issue from the review
4. Verify no regressions — check that related functionality still works

### 3f. Update progress
After each fix is applied and verified, update the fix plan file to mark it as done:
- Add `**Status: DONE**` to the completed fix
- If a fix fails or is skipped, add `**Status: SKIPPED - <reason>**`

### 3g. Ask before continuing
After each fix, show:
- What was changed (brief diff summary)
- Verification result
- Ask user: continue to next fix, skip, or stop?

## Step 4: Summary

After all selected fixes are applied, create a summary file at `./claude-context/$ARGUMENTS/fix-summary.md`:

```markdown
# Fix Summary for branch: <branch-name>
Date: <current date>

## Applied Fixes
| # | Title | Severity | Repo | Status |
|---|-------|----------|------|--------|
| 1 | ...   | Critical | ...  | DONE   |
| 2 | ...   | High     | ...  | DONE   |

## Skipped Fixes
| # | Title | Reason |
|---|-------|--------|

## Remaining Issues
- <any issues not addressed>

## Recommendation
- <whether the branch is now production-ready>
- <any manual steps still needed>
```

Inform the user:
- How many fixes were applied vs skipped
- Whether branch is now ready for re-review or merge
- Suggest running `/review-branch $ARGUMENTS` again if critical fixes were applied
